## Continuous Integration

GitHub Actions CI is defined in `.github/workflows/ci.yml`.

The workflow runs on pushes to `main` and on all pull requests, and it validates:

- `npm run build`
- `npm run test:coverage -- --runInBand`
- `npm run cucumber`

Coverage and Cucumber JSON reports are uploaded as workflow artifacts.

## Development Process

This project follows an AI-assisted development workflow that spans from product conception through deployment. Each phase leverages purpose-fit AI tools to accelerate delivery while maintaining quality.

---

### 1. Product Concept and Feature Prioritization

#### a. Domain Modeling in Miro with Sidekick (Claude Sonnet 3.7)

The data model and domain relationships are designed interactively using Miro's AI Sidekick (powered by Claude Sonnet 3.7). The conversation-driven modeling session produces a structured Entity Relationship Diagram exported as:

- **Output:** [`documentation/design/erd/job_tracker_erd.md`](documentation/design/erd/job_tracker_erd.md)

#### b. Feature Prioritization using WSJF Scoring (Claude Sonnet 3.7)

A prioritized product backlog is generated by prompting Miro Sidekick to evaluate candidate features using **Weighted Shortest Job First (WSJF)** scoring — weighing business value, time criticality, risk reduction, and job size to surface the highest-impact work first.

- **Output:** [`documentation/design/feature-backlog.md`](documentation/design/feature-backlog.md)

---

### 2. Development Cycle

Each feature follows a consistent loop from specification through delivery:

#### a. Generate Feature Requirements in Gherkin (Claude Sonnet 3.7)

Gherkin-formatted acceptance criteria are requested from Miro Sidekick for the next prioritized feature. This produces human-readable, executable specifications that bridge product intent and test automation.

#### b. Import Gherkin Feature into Cucumber

The generated `.feature` files are added to `e2e/features/`, making them immediately executable as BDD scenarios via the Cucumber test runner.

---

### 3. Feature Review and Plan Prompt (GitHub Copilot — Claude Sonnet 4.6)

Before implementation begins, GitHub Copilot (using Claude Sonnet 4.6) is asked to review the Gherkin feature and produce a detailed implementation plan prompt. This step surfaces design considerations, identifies edge cases, and aligns the implementation approach with the existing codebase patterns.

---

### 4. Execute the Plan Prompt (GitHub Copilot — Claude Sonnet 4.6)

The implementation plan prompt from step 3 is executed with GitHub Copilot (Claude Sonnet 4.6) to generate the feature code — components, services, models, and supporting tests — directly within VS Code.

---

### 5. Manual Testing

The implemented feature is exercised manually in the browser to validate behaviour against the acceptance criteria, catch visual or UX regressions, and verify edge cases not covered by automated tests.

---

### 6. Pull Request and Copilot Code Review

A pull request is opened on GitHub. **GitHub Copilot (Web)** is requested as a reviewer to perform an automated code review. Any flagged issues — correctness, security, style, or test coverage — are resolved before merge.

---

### 7. Merge and Deploy

Once all review issues are resolved and CI passes (build, unit tests with coverage, Cucumber BDD scenarios), the branch is merged to `main` and deployed.
